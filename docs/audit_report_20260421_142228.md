## 三、当前发现

### ❗ Critical

#### C1. 前端当前无法构建，属于交付阻塞

- 证据：`npm run build` 失败
- 受影响文件：
  - `frontend/src/charts/RealtimeCurveChart.tsx:29-39`
  - `frontend/src/charts/RiskRadarChart.tsx:92-112`
  - `frontend/src/charts/SocHeatmap.tsx:11-17`
  - `frontend/src/components/TopIndicators.tsx:21`
- 影响：
  - 当前前端不能生成可部署构建产物
  - 类型系统已经失守，说明图表组件与领域类型没有收敛到同一契约
- 修复方向：
  - 先消除 TypeScript 编译错误，再继续判断页面行为
  - 优先统一图表 props 的字面量类型、去除重复对象键、为 heatmap 数据补充明确 tuple 类型、修正 `TopIndicators` 对不存在字段的访问

#### C2. 设备模块前端请求路径仍指向 `/device/*`，与后端 `/devices/*` 已不兼容

- 证据：
  - 前端：`frontend/src/api/device.ts:6-17`
  - 后端：`backend/api-service/src/main/java/com/battery/platform/api/controller/DeviceController.java:14,20-52`
  - 规范：`docs/api-spec.yaml:272-368` 使用 `/devices/*`
- 影响：
  - 设备台账、拓扑树、量测映射、创建单元/簇接口会直接 404
  - 旧报告中“后端路径已修复”的改动尚未同步到前端，形成回归缺陷
- 修复方向：
  - 将前端设备 API 全量改为 `/devices/*`
  - 联动核对 `DeviceLedger` 和 `deviceSlice` 的请求是否与返回结构一致

#### C3. 告警列表前后端分页协议不兼容，前端会把有数据的接口当成空列表

- 证据：
  - 前端 API 类型：`frontend/src/api/alarm.ts:6-7`
  - 前端消费逻辑：`frontend/src/store/slices/alarmSlice.ts:46-49`
  - 后端返回类型：`backend/api-service/src/main/java/com/battery/platform/api/controller/AlarmController.java:23-30`
- 问题细节：
  - 前端假设后端返回 `{ items, total, page, pageSize }`
  - 后端实际返回 Spring `Page<AlarmEvent>`，字段形状是 `content/totalElements/number/size/...`
- 影响：
  - 告警表格可能始终为空
  - 未读数计算始终错误
- 修复方向：
  - 二选一统一协议：
    - 后端封装自定义分页 DTO
    - 或前端改为读取 Spring Page 结构
  - 修复后补一条真实的告警列表集成验证

### ⚠️ Major

#### M1. 登录响应契约已漂移，前端声明的 `user` 对象后端根本不返回

- 证据：
  - 前端类型：`frontend/src/types/auth.ts:6-17`
  - 前端落库：`frontend/src/store/slices/authSlice.ts:46-50`
  - 后端返回：`backend/api-service/src/main/java/com/battery/platform/api/service/AuthService.java:46-50`
  - 控制器直通：`backend/api-service/src/main/java/com/battery/platform/api/controller/AuthController.java:17-34`
- 问题细节：
  - 前端期望 `LoginResponse = { token, refreshToken, user }`
  - 后端实际返回 `{ token, refreshToken, username, displayName }`
- 影响：
  - `state.user` 被置为 `undefined`
  - 任何依赖用户信息、角色或权限的前端逻辑都没有可信输入
- 修复方向：
  - 统一登录响应 DTO
  - 若前端确实需要用户对象，应由后端显式返回 `id/username/displayName/roles/permissions`

#### M2. 驾驶舱领域模型在前后端之间已经严重漂移

- 证据：
  - 后端 DTO：`backend/api-service/src/main/java/com/battery/platform/api/dto/DashboardOverviewDTO.java:7-16`
  - 前端类型：`frontend/src/types/battery.ts:19-83`
  - 前端组件：`frontend/src/components/TopIndicators.tsx:21-26`
- 问题细节：
  - 后端只返回 8 个左右字段
  - 前端 `DashboardOverview` 声明了大量 PRD 级字段，如 `realSoh`、`riskRadar`、`swapSuggestions`、`sohTrend` 等
  - `TopIndicators` 甚至读取了类型中都不存在的 `overview.totalDays`
- 影响：
  - 当前不只是“接口字段少”，而是领域模型已经分裂成三套：PRD、前端类型、后端 DTO
  - 这也是当前构建失败的直接原因之一
- 修复方向：
  - 以 `api-spec.yaml` 或新的共享 DTO 为唯一真源
  - 先统一 `DashboardOverview` 最小闭环字段，再让页面逐步脱离 mock

#### M3. 问诊室接口与前端类型也已失配，后续一旦接真数据会直接破

- 证据：
  - 后端评估 DTO：`backend/api-service/src/main/java/com/battery/platform/api/dto/ClinicAssessmentDTO.java`
  - 后端实现：`backend/api-service/src/main/java/com/battery/platform/api/service/ClinicService.java:18-60`
  - 前端类型：`frontend/src/types/battery.ts:120-149`
  - 前端 API：`frontend/src/api/clinic.ts:4-7`
- 问题细节：
  - 前端期待 `AssessmentItem` 包含 `realSoh`、`theorySoh`、`risks`、`lastUpdateTime`
  - 后端实际只返回 `soh`、`riskLevel`、`highRiskItemCount` 等另一套字段
  - 端点分析前端期待 `curves` 与若干端点指标，后端返回的是 `endpoints[] + overallRisk`
- 影响：
  - 问诊室当前 UI 之所以没炸，主要是因为页面还大量使用 mock 数据
  - 一旦接入真数据，会直接出现类型和渲染错误
- 修复方向：
  - 先确定问诊室接口的真实 wire shape
  - 前后端必须围绕同一 DTO 重建

#### M4. 多个核心页面仍以硬编码 UI 和 mock 数据为主，功能完整性不足

- 证据：
  - 驾驶舱：`frontend/src/pages/Dashboard/index.tsx:34-42,75-158`
  - 问诊室概览：`frontend/src/pages/Clinic/Overview.tsx:33-118`
  - 问诊室明细：`frontend/src/pages/Clinic/AssessmentDetail.tsx`
  - 后端问诊：`backend/api-service/src/main/java/com/battery/platform/api/service/ClinicService.java:18-60`
  - 运维：`backend/api-service/src/main/java/com/battery/platform/api/service/OmService.java:54-68`
- 影响：
  - 页面外观存在，但关键业务图表、统计、告警与问诊结论并未形成真实闭环
  - 旧报告中“核心服务仍是占位”的判断在当前版本仍然部分成立
- 修复方向：
  - 先完成数据契约统一
  - 再按页面逐个替换 mock 与回退逻辑

#### M5. 告警消费链路仍无重试/DLQ，失败消息会被吞掉

- 证据：`backend/api-service/src/main/java/com/battery/platform/api/alarm/AlarmConsumer.java:15-22`
- 问题细节：
  - 消费异常仅打印日志，不抛出、不重试、不转死信
- 影响：
  - 告警匹配或持久化失败时，消息会静默丢失
- 修复方向：
  - 接入 Spring Kafka 错误处理器、重试策略和死信主题
  - 把失败样本纳入运维监控

#### M6. 前端认证 token 仍存储在 `localStorage`

- 证据：
  - `frontend/src/api/client.ts:10-13,34-43`
  - `frontend/src/store/slices/authSlice.ts:15-16,25-26,33-34`
- 影响：
  - 一旦发生 XSS，Access Token 与 Refresh Token 都可被直接读取
- 修复方向：
  - 至少迁移 Refresh Token 到 `httpOnly` Cookie
  - Access Token 应改为内存态或更短生命周期方案

### ℹ️ Minor

#### N1. 网关仍保留默认 JWT 密钥回退，存在安全与一致性隐患

- 证据：`backend/gateway/src/main/resources/application.yml:14-15`
- 问题细节：
  - `api-service` 要求 `JWT_SECRET` 必填
  - `gateway` 仍允许默认回退字符串
- 影响：
  - 若环境变量漏配，网关可能以默认密钥启动，和下游服务验签策略不一致
- 修复方向：
  - 取消默认值，改成与 `api-service` 相同的强制注入策略

#### N2. `SystemLogController` 只支持“action+时间范围”联合过滤，不支持单独时间范围

- 证据：
  - 控制器：`backend/api-service/src/main/java/com/battery/platform/api/controller/SystemLogController.java:31-38`
  - 规范：`docs/api-spec.yaml:447-463`
- 影响：
  - 文档允许按时间查询，但当前实现只有在 `action/startTime/endTime` 同时存在时才会走时间范围分支
- 修复方向：
  - 为时间范围单独提供 repository/query 路径

#### N3. `RedisConfig` 虽已收敛，但仍依赖 `activateDefaultTyping`

- 证据：`backend/api-service/src/main/java/com/battery/platform/api/config/RedisConfig.java:25-30`
- 影响：
  - 风险已明显低于旧实现，但仍建议仅在确有多态对象需求时保留
- 修复方向：
  - 评估是否可改为显式 DTO 序列化，进一步收缩反序列化面

## 七、结论

当前版本不是“旧报告问题原样未修”，而是处于“部分高风险问题已修复，但前后端契约重新失配”的状态。最紧急的问题已经从“后端缺少控制器”转移为“前端无法构建”和“接口模型没有统一真源”。

建议后续整改顺序不要再按旧报告逐项照单全收，而应先完成三件事：

1. 恢复前端构建
2. 统一认证、设备、告警、驾驶舱、问诊室 DTO
3. 再替换残留 mock/回退逻辑

在这三项完成之前，继续追加页面开发会持续放大契约债务。
