# 四阶段执行路线图

> 目标：先把 MVP 演示做稳，再平滑接客户真实设备，最后补齐故障诊断与阻抗谱能力。

## 1. 阶段一：MVP 演示

目标：
- 服务器上一键展示现有版本
- 前端、后端、算法服务能独立启动
- 默认走 demo 数据，不依赖客户现场设备

边界：
- 不接真实设备
- 不承诺阻抗谱/EIS
- 不追求算法精度，只保留稳定接口

交付物：
- Docker Compose 演示编排
- 演示部署文档
- 诊断总览、告警、问诊室可访问

## 2. 阶段二：客户真实设备接入

目标：
- 接入客户 BMS / PCS / EMS / 边缘网关
- 统一数据模型、点表、Topic 和落库链路

边界：
- 设备协议适配与平台业务逻辑解耦
- 先做遥测接入，不先做双向控制

交付物：
- 点表模板
- 协议映射规范
- MQTT/Kafka/InfluxDB 接入链路

## 3. 阶段三：故障诊断平台

目标：
- 把现有健康评估升级为“故障诊断”
- 形成诊断结论、证据链、处置建议、历史追溯

边界：
- 诊断结果必须可解释、可追踪
- 算法能力先以可用为主，再逐步替换为更强模型

交付物：
- 诊断域数据结构
- 故障案例库
- 诊断结果 API
- 前端诊断工作台

## 4. 阶段四：阻抗谱 / EIS 算法接入

目标：
- 支持阻抗谱数据采集、存储、可视化和诊断
- 提供 Nyquist / Bode 等基础分析入口

边界：
- EIS 数据单独建模，不和普通遥测混写
- 算法先做 baseline，再逐步升级

交付物：
- EIS 数据契约
- EIS 算法占位服务
- EIS 可视化页面

## 5. 当前执行顺序

1. 先完成阶段一的可交付状态。
2. 同步建立阶段三、四的契约骨架，避免后续返工。
3. 阶段二等客户设备资料到位后再落协议适配。
