## 二、安全漏洞审计 🟠

### 2.1 [高危] Redis 反序列化漏洞

**文件**：`RedisConfig.java`，第24行

```java
// 当前危险配置
mapper.activateDefaultTyping(
    LaissezFaireSubTypeValidator.instance,
    ObjectMapper.DefaultTyping.NON_FINAL  // 已知RCE攻击向量
);
```

`DefaultTyping.NON_FINAL` + `LaissezFaireSubTypeValidator` 允许反序列化任意 Java 类型，是著名的 Jackson 反序列化 RCE 攻击入口。

**修复方案**：
```java
PolymorphicTypeValidator ptv = BasicPolymorphicTypeValidator.builder()
    .allowIfSubType("com.battery.platform")
    .build();
mapper.activateDefaultTyping(ptv, ObjectMapper.DefaultTyping.NON_FINAL);
```

---

### 2.2 [高危] 前端 Token 存储于 localStorage（XSS 可窃取）

**文件**：`api/client.ts`、`store/slices/authSlice.ts`

JWT Access Token 和 Refresh Token 均存储在 `localStorage`，可被任何 XSS 脚本读取。

**修复方案**：Access Token 使用内存存储（Redux State），Refresh Token 使用 `httpOnly` Cookie。

---

### 2.3 [中危] 全局异常处理泄露内部信息

**文件**：`GlobalExceptionHandler.java`，第34行

```java
// 直接将异常 message 返回给客户端，可能包含SQL、堆栈路径等敏感信息
"message", e.getMessage()
```

**修复方案**：生产环境的 500 错误应返回通用消息，内部异常信息只记录日志。

---

### 2.4 [中危] 算法引擎无任何认证

**文件**：`algorithm/app.py`

FastAPI 服务完全无认证（无 API Key、无 JWT），任何能访问该端口的请求都可调用算法接口。网络隔离是唯一防线，不符合纵深防御原则。

**修复方案**：添加共享密钥 Header 验证（`X-Internal-Token`）或 mTLS。

---

### 2.5 [中危] AuthService 刷新 Token 未区分 Token 类型

**文件**：`AuthService.java`，第51-63行

`refresh()` 方法接受任何有效 JWT（包括 Access Token），因为 Access Token 和 Refresh Token 用完全相同的密钥和 claims 生成（第42-43行两次调用同一 `generateToken`）。攻击者可用未过期的 Access Token 无限获取新 Token。

**修复方案**：在 `generateToken` 中加入 `type` claim（`ACCESS`/`REFRESH`），刷新时校验 `type == REFRESH`。

---

### 2.6 [低危] SLF4J 日志参数格式问题

**文件**：`JwtAuthenticationFilter.java`，第52行

```java
logger.warn("JWT 验证失败: {}", e.getMessage());
// 建议改为以下形式以保留完整堆栈
logger.warn("JWT 验证失败", e);
```

## 三、代码质量审计 🟡

### 3.1 API 路径命名不一致

| 问题 | 规范定义 | 实际实现 |

### 3.2 DeviceController 响应格式不统一

`DeviceController` 直接返回实体对象（`ResponseEntity<List<StationNode>>`），未包裹 `ApiResponse<T>`，而其他所有 Controller 均使用 `ApiResponse.ok()`。前端 `client.ts` 的响应拦截器将无法正确解包此端点的响应。

---

### 3.3 前端 Axios 拦截器逻辑错误（Dead Code）

**文件**：`api/client.ts`，第17-35行

```typescript
// 第22-27行：处理 code=200 的响应
if (payload.code === 200) { return payload.data; }

// 第29-34行：条件完全相同，第二段永远不会执行！
// 算法引擎返回 { code: 0, data: ... } 格式永远无法被正确解包
if (payload.code === 0) { return payload.data; }  // Dead code
```

**修复方案**：
```typescript
if (payload.code === 200 || payload.code === 0) {
  return payload.data;
}
```

---

### 3.4 ClinicService 风险枚举超出 PRD 定义

**文件**：`ClinicService.java`，第13行

```java
String[] riskLevels = {"LOW", "MEDIUM", "HIGH", "CRITICAL"};
```

PRD 仅定义三级（`HIGH`/`MEDIUM`/`LOW`），`CRITICAL` 为自创枚举。会导致前端 `SafetyAssessment` 的风险权重映射失效，且安全评估排序逻辑（high=3, medium=2, low=1）错误。

---

### 3.5 WebSocket URL 协议替换逻辑冗余

**文件**：`useWebSocket.ts`，第20行

```typescript
const wsUrl = url.replace(/^ws:/, window.location.protocol === 'https:' ? 'wss:' : 'ws:');
```

在 HTTP 环境下将 `ws:` 替换为 `ws:`（无实际变化），逻辑冗余但不影响功能。建议明确处理 `wss://` 降级场景。

---

### 3.6 `AlarmEvent` 实体缺少时区配置

`AlarmEvent.createdAt` 等字段为 `LocalDateTime`，在跨时区部署（国产信创 UOS Linux）时可能因 JVM 时区与数据库时区不一致产生时间漂移。建议统一 UTC 存储，并显式配置 `@Column(columnDefinition = "DATETIME(6)")`.

---

### 3.7 `UserNode` 缺少主键生成策略

**文件**：`UserNode.java`

```java
@Id
private String id;  // 无 @GeneratedValue，需业务层手动赋 UUID
```

`AuthService.generateToken()` 使用 `user.getId()` 作为 JWT Subject，若 `id` 为 null 则生成的 Token 无效，且不会抛出异常（JWT subject 为字符串 "null"）。

## 四、性能问题审计 🟡

### 4.1 告警规则每次从 Neo4j 全量加载

**文件**：`AlarmRuleMatcher.java`，第23行

```java
List<AlarmRuleNode> rules = alarmRuleNodeRepository.findByEnabledTrue();
```

每条 Kafka 消息到达时都全量查询图数据库。PRD 明确要求使用 Redis `alarm_rule:` 前缀缓存，但未实现。高频告警场景下将对 Neo4j 产生严重压力。

---

### 4.2 Kafka Consumer 未配置并发数

**文件**：`KafkaConfig.java`

`ConcurrentKafkaListenerContainerFactory` 未设置 `setConcurrency()`，默认并发数为1，无法并行处理多分区告警消息。

---

### 4.3 算法服务缺少启动时健康检查

**文件**：`app.py`

所有算法模块采用路由内 lazy import（如 `from core.derive import calculate_soh`），若模型文件缺失或依赖未安装，错误将在首次请求时才暴露，而非服务启动时。建议在 `startup` 事件中预加载并校验模型文件。

---

### 4.4 `useWebSocket` Hook 依赖项稳定性问题

**文件**：`useWebSocket.ts`，第47行

`onMessage` 作为 `useCallback` 依赖项，若调用方每次渲染传入新函数引用（未 `useCallback` 封装），将导致 WebSocket 反复重建连接。
